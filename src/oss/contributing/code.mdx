코드 기여는 언제나 환영합니다! 버그 수정, 기능 추가, 성능 개선 등 여러분의 기여는 수천 명의 개발자들에게 더 나은 개발자 경험을 제공하는 데 도움이 됩니다.

<Note>
    대규모 **새로운 기능이나 리팩토링**을 제출하기 전에 먼저 [포럼](https://forum.langchain.com/)에서 아이디어를 논의해 주세요. 이를 통해 프로젝트 목표와의 일치를 보장하고 중복 작업을 방지할 수 있습니다.

    이는 버그 수정이나 소규모 개선에는 적용되지 않으며, pull request를 통해 직접 기여할 수 있습니다. PR 설명에 관련 이슈를 반드시 링크하세요. <Tooltip tip="(예: `Fixes #123`)">closing keyword</Tooltip>를 사용하여 PR이 병합될 때 이슈를 자동으로 닫을 수 있습니다.

    새로운 통합은 [통합 가이드라인](/oss/contributing#add-a-new-integration)을 따라야 합니다.
</Note>

## 철학

모든 코드 기여에 대해 다음 핵심 원칙을 따르는 것을 목표로 합니다:

    <Card title="하위 호환성" icon="shield" href="#backwards-compatibility" arrow>
        안정적인 public interface를 유지하고 breaking change를 피합니다

        모든 변경사항은 정확성을 검증하고 회귀를 방지하기 위한 포괄적인 테스트를 포함해야 합니다
    </Card>

    <Card title="코드 품질" icon="star" href="#code-quality-standards" arrow>
        일관된 스타일, 문서화 및 아키텍처 패턴을 따릅니다

    </Card>
</CardGroup>

## 시작하기

### 빠른 수정: 버그 수정 제출하기

간단한 버그 수정의 경우 즉시 시작할 수 있습니다:

<Steps>
    <Step title="repository fork하기">
        :::python
        [LangChain](https://github.com/langchain-ai/langchain) 또는 [LangGraph](https://github.com/langchain-ai/langgraph) repo를 <Tooltip tip="조직 계정으로 fork하면 관리자가 편집할 수 없습니다.">개인 GitHub 계정</Tooltip>으로 fork하세요
        :::
        :::js
        [LangChain](https://github.com/langchain-ai/langchainjs) 또는 [LangGraph](https://github.com/langchain-ai/langgraphjs) repo를 개인 GitHub 계정으로 fork하세요
        :::
    </Step>

    <Step title="Clone 및 설정">
        :::python

        ```bash
        git clone https://github.com/your-username/name-of-forked-repo.git

        # For instance, for LangChain:
        git clone https://github.com/parrot123/langchain.git

        # For LangGraph:
        git clone https://github.com/parrot123/langgraph.git
        ```

        ```bash
        # Inside your repo, install dependencies
        uv sync --all-groups
        ```

        이전에 설치하지 않았다면 [`uv`](https://docs.astral.sh/uv/)를 설치해야 합니다.
        :::
        :::js

        ```bash
        git clone https://github.com/your-username/name-of-forked-repo.git

        # For instance, for LangChain:
        git clone https://github.com/parrot123/langchainjs.git

        # For LangGraph:
        git clone https://github.com/parrot123/langgraphjs.git
        ```

        ```bash
        # Inside your repo, install dependencies
        pnpm install
        # Create a build for all packages to resolve workspace dependencies
        pnpm build
        ```
        :::
    </Step>

    <Step title="branch 생성">
        ```bash
        git checkout -b your-username/short-bugfix-name
        ```
    </Step>

    <Step title="변경사항 작성">
        [코드 품질 표준](#code-quality-standards)을 따르면서 버그를 수정하세요
    </Step>

    <Step title="테스트 추가">
        수정 없이는 실패하는 [unit test](#test-writing-guidelines)를 포함하세요. 이를 통해 버그가 해결되었는지 확인하고 회귀를 방지할 수 있습니다
    </Step>

    :::js
    <Step title="빌드 실행">
        패키지가 여전히 제대로 빌드되는지 확인하기 위해 빌드 명령을 실행하세요
        ```bash
        pnpm build
        # or build a specific workspace package
        pnpm --filter @langchain/core build
        ```
    </Step>
    :::

    <Step title="테스트 실행">
        PR을 제출하기 전에 모든 테스트가 로컬에서 통과하는지 확인하세요

        :::python
        ```bash
        make lint
        make test

        # For bugfixes involving integrations, also run:
        make integration_tests
        ```
        :::
        :::js
        ```bash
        pnpm lint
        pnpm test

        # For bugfixes involving integrations, also run:
        pnpm test:int

        # Or run tests in a specific workspace package
        cd libs/langchain-core
        pnpm test
        pnpm lint

        # Or run tests for a specific package from the root of the repo
        pnpm --filter @langchain/core test
        pnpm --filter @langchain/core lint
        ```
        :::
    </Step>

    <Step title="pull request 제출">
        제공된 PR 템플릿을 따르세요. 해당되는 경우 [closing keyword](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/linking-a-pull-request-to-an-issue#linking-a-pull-request-to-an-issue-using-a-keyword)를 사용하여 수정하는 이슈를 참조하세요 (예: `Fixes #123`).
    </Step>
</Steps>

## 개발 워크플로우

### 테스트 요구사항

<Info>
    디렉토리는 작업 중인 패키지를 기준으로 합니다.
</Info>

모든 코드 변경사항은 포괄적인 테스트를 포함해야 합니다.

<Steps>
    <Step title="Unit test">
        :::python
        **위치**: `tests/unit_tests/`
        :::
        :::js
        **위치**: `src/tests/FILENAME_BEING_TESTED.test.ts`
        :::

        **요구사항**:
        - 네트워크 호출 금지
        - edge case를 포함한 모든 코드 경로 테스트
        - 외부 의존성에 대해 mock 사용

        :::python
        ```bash
        make test
        # Or directly:
        uv run --group test pytest tests/unit_tests
        ```
        :::
        :::js
        ```bash
        # Run the entire test suite
        pnpm test
        # Or run a specific test file
        pnpm test src/tests/FILENAME_BEING_TESTED.test.ts
        # Or run a specific test function
        pnpm test -t "the test that should be run"
        ```
        :::
    </Step>

    <Step title="Integration test">
        Integration test는 외부 서비스/provider API에 대한 액세스가 필요하며 (비용이 발생할 수 있음) 따라서 기본적으로 실행되지 않습니다.

        모든 코드 변경사항이 integration test를 필요로 하는 것은 아니지만, 검토 프로세스의 일부로 integration test를 별도로 요구/실행할 것임을 유념하세요.

        :::python
        **위치**: `tests/integration_tests/`
        :::
        :::js
        **위치**: `src/tests/FILENAME_BEING_TESTED.int.test.ts`
        :::

        **요구사항**:
        - 외부 서비스와의 실제 통합 테스트
        - API key에 환경 변수 사용
        - 자격 증명을 사용할 수 없는 경우 gracefully skip

        :::python
        ```bash
        make integration_tests
        ```
        :::
        :::js
        ```bash
        pnpm test:int
        ```
        :::
    </Step>

    <Step title="테스트 품질 체크리스트">
        - 코드가 손상되면 테스트가 실패함
        - Edge case 및 오류 조건이 테스트됨
        - fixture 및 mock의 적절한 사용
    </Step>
</Steps>

### 코드 품질 표준

품질 요구사항:

<Tabs>
    <Tab title="Type hint">
        :::python
        **필수**: 모든 함수에 대한 완전한 type annotation

        ```python
        def process_documents(
            docs: list[Document],
            processor: DocumentProcessor,
            *,
            batch_size: int = 100
        ) -> ProcessingResult:
            """Process documents in batches.

            Args:
                docs: List of documents to process.
                processor: Document processing instance.
                batch_size: Number of documents per batch.

            Returns:
                Processing results with success/failure counts.
            """
        ```
        :::
        :::js
        **필수**: 모든 함수에 대한 완전한 type

        ```typescript
        function processDocuments(
            docs: Document[],
            processor: DocumentProcessor,
            batchSize: number = 100
        ): ProcessingResult {
            // ...
        }
        ```
        :::
    </Tab>

    <Tab title="문서화">
        :::python
        **필수**: 모든 public 함수에 대한 [Google-style docstring](https://google.github.io/styleguide/pyguide.html)
        :::
        :::js
        **필수**: 모든 export된 함수 및 interface에 대한 [JSDocs](https://jsdoc.app/about-getting-started)

        ```typescript
        /**
         * Document processing instance.
         */
        interface FooDocumentProcessor {
            /**
             * Process documents in batches.
             *
             * @param docs - List of documents to process.
             * @returns Processing results with success/failure counts.
             */
            process(docs: Document[]): ProcessingResult;
        }

        /**
         * Process documents in batches.
         *
         * @param docs - List of documents to process.
         * @param processor - Document processing instance.
         * @param batchSize - Number of documents per batch.
         * @returns Processing results with success/failure counts.
         */
        export function processDocuments(
            docs: Document[],
            processor: DocumentProcessor,
            batchSize: number = 100
        ): ProcessingResult {
            // ...
        }
        ```
        :::

        - 모든 parameter 및 return value 문서화
        - 복잡한 함수에 대한 사용 예제 포함
        - 발생하는 exception 문서화
        - "무엇"보다 "왜"에 초점

    </Tab>

    <Tab title="코드 스타일">
        :::python
        **자동화**: [`ruff`](https://docs.astral.sh/ruff/)를 통한 formatting 및 linting

        ```bash
        make format  # Apply formatting
        make lint    # Check style and types
        ```
        :::
        :::js
        **자동화**: Formatting 및 linting:

        ```bash
        pnpm lint    # Check style and types
        pnpm format  # Apply formatting
        ```
        :::

        **표준**:
        - 설명적인 변수명
        - 복잡한 함수 분해 (20줄 미만 목표)
        - 코드베이스의 기존 패턴 따르기
    </Tab>
</Tabs>

:::python
### 수동 formatting 및 linting

<Note>
    코드 formatting 및 linting은 CI/CD를 통해 강제됩니다. 코드가 검사를 통과하도록 커밋하기 전에 이러한 명령을 실행하세요.
</Note>

Formatting 및 linting 실행:

<Steps>
    <Step title="코드 format">
        ```bash
        make format
        ```
    </Step>

    <Step title="Linting 검사 실행">
        ```bash
        make lint
        ```
    </Step>

    <Step title="변경사항 확인">
        두 명령 모두 커밋하기 전에 해결해야 할 formatting 또는 linting 문제를 표시합니다.
    </Step>
</Steps>

## 기여 가이드라인

### 하위 호환성

<Warning>
    중요한 보안 수정을 제외하고 public API에 대한 breaking change는 허용되지 않습니다.

    major 버전 릴리스에 대한 자세한 내용은 [버전 관리 정책](/oss/versioning)을 참조하세요.
</Warning>

호환성 유지:

<AccordionGroup>
    <Accordion title="안정적인 interface">
        **항상 보존**:
        - 함수 signature 및 parameter 이름
        - Class interface 및 method 이름
        - Return value 구조 및 type
        - Public API의 import 경로
    </Accordion>

    <Accordion title="안전한 변경">
        **허용되는 수정**:
        :::python
        - 새로운 optional parameter 추가
        :::
        :::js
        - 새로운 optional parameter/type parameter 추가
        :::
        - class에 새로운 method 추가
        - 동작을 변경하지 않고 성능 개선
        - 새로운 module 또는 함수 추가
    </Accordion>

    <Accordion title="변경하기 전에">
        - **기존 사용자 코드가 손상될까요?**
        - 대상이 public인지 확인
        :::python
        - 필요한 경우 `__init__.py`에서 export되는지 확인
        :::
        - 테스트에 기존 사용 패턴이 있는지 확인
    </Accordion>
</AccordionGroup>

### 버그 수정

버그 수정 기여의 경우:

<Steps>
    <Step title="이슈 재현">
        버그를 보여주는 최소한의 테스트 케이스를 생성하세요. 관리자 및 다른 기여자가 추가 설정이나 수정 없이 이 테스트를 실행하고 실패를 확인할 수 있어야 합니다
    </Step>

    <Step title="실패하는 테스트 작성">
        수정 없이는 실패할 unit test를 추가하세요
    </Step>

    <Step title="수정 구현">
        문제를 해결하는 데 **필요한 최소한의 변경**을 수행하세요
    </Step>

    <Step title="수정 확인">
        테스트가 통과하고 회귀가 발생하지 않는지 확인하세요
    </Step>

    <Step title="변경사항 문서화">
        동작이 변경되면 docstring을 업데이트하고, 복잡한 로직에 주석을 추가하세요
    </Step>
</Steps>

### 새로운 기능

새로운 기능에 대해 높은 기준을 유지하는 것을 목표로 합니다. 일반적으로 새로운 core abstraction, infra 변경, 의존성 변경 또는 새로운 agent/chain에 대한 외부 기여자의 기여는 이에 대한 명확한 필요성을 보여주는 기존 이슈 없이는 수락하지 않습니다.

일반적으로 기능 기여 요구사항은 다음과 같습니다:

<Steps>
    <Step title="설계 논의">
        다음을 설명하는 이슈를 열어주세요:
        - 해결하려는 문제
        - 제안된 API 설계
        - 예상 사용 패턴
    </Step>

    <Step title="구현">
        - 기존 코드 패턴 따르기
        - 포괄적인 테스트 및 문서 포함
        - 보안 영향 고려
    </Step>

    <Step title="통합 고려사항">
        - 기존 기능과 어떻게 상호작용하나요?
        - 성능에 영향이 있나요?
        - 새로운 의존성이 도입되나요?

        보안 취약점이나 보고로 이어질 가능성이 있는 기능은 거부합니다.
    </Step>
</Steps>

### 보안 가이드라인

<Warning>
    보안이 가장 중요합니다. 취약점이나 안전하지 않은 패턴을 절대 도입하지 마세요.
</Warning>

보안 체크리스트:

<AccordionGroup>
    <Accordion title="입력 검증">
        - 모든 사용자 입력 검증 및 sanitize
        - template 및 query의 데이터를 적절히 escape
        :::python
        - 사용자 데이터에 `eval()`, `exec()` 또는 `pickle`을 절대 사용하지 마세요. 이는 임의 코드 실행 취약점으로 이어질 수 있습니다
        :::
        :::js
        - `eval()`을 절대 사용하지 마세요. 이는 임의 코드 실행 취약점으로 이어질 수 있습니다
        :::
    </Accordion>

    <Accordion title="오류 처리">
        - 특정 exception type 사용
        - 오류 메시지에 민감한 정보 노출 금지
        - 적절한 리소스 정리 구현
    </Accordion>

    <Accordion title="의존성">
        - hard dependency 추가 피하기
        - Optional dependency를 최소화
        - 보안 문제에 대해 third-party 패키지 검토
    </Accordion>
</AccordionGroup>

### 테스트 작성 가이드라인

효과적인 테스트를 작성하기 위해 따라야 할 몇 가지 좋은 관행이 있습니다:

:::python
- Docstring에서 자연어를 사용하여 테스트 설명
- 설명적인 변수명 사용
- assertion을 철저히 작성
:::
:::js
- 테스트 중인 component를 설명하는 `describe` 블록으로 테스트 캡슐화
- 테스트 이름을 설명하기 위해 자연어 사용
- Assertion을 철저히 작성
- 합리적인 크기의 데이터 객체에만 snapshot 사용
:::

<Tabs>
    <Tab title="Unit test">
        :::python
        ```python
        def test_document_processor_handles_empty_input():
            """Test processor gracefully handles empty document list."""
            processor = DocumentProcessor()

            result = processor.process([])

            assert result.success
            assert result.processed_count == 0
            assert len(result.errors) == 0
        ```
        :::
        :::js
        ```typescript
        describe("DocumentProcessor", () => {
            it("Should handle empty document list", () => {
                const processor = new DocumentProcessor();
                const result = processor.process([]);

                expect(result.success).toBe(true);
                expect(result.processedCount).toBe(0);
                expect(result.errors).toHaveLength(0);
            });
        });
        ```
