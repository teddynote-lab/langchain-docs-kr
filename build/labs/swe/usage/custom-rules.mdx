# Custom Rules

Custom rules를 사용하면 저장소 루트의 markdown 파일을 통해 Open SWE에 프로젝트별 컨텍스트와 가이드라인을 제공할 수 있습니다.

## 시작하기

시작하는 가장 쉬운 방법은 Open SWE가 직접 이 파일을 작성하도록 하는 것입니다! Open SWE UI는 "Generate `AGENTS.md`" 빠른 작업을 제공하며, 이는 미리 정의된 프롬프트를 입력란에 삽입합니다. 이 프롬프트는 에이전트로부터 잘 포맷된 `AGENTS.md` 파일을 생성하도록 설계되었습니다.

![Quick Action Generate AGENTS.md](/images/quick_action_generate_agents_md.png)

## `AGENTS.md`란 무엇인가요?

`AGENTS.md`는 프로젝트의 다음 사항을 Open SWE에 알려주는 markdown 파일입니다:

- 코딩 표준 및 규칙
- 저장소 구조 및 아키텍처
- 의존성 및 설치 절차
- 테스팅 프레임워크 및 관행
- Pull request 포맷 요구사항

## Custom Rules를 사용하는 이유는?

Custom rules는 Open SWE가 다음을 수행하는 데 도움을 줍니다:

- 프로젝트의 특정 규칙 준수
- 코드베이스 아키텍처 이해
- 올바른 패키지 매니저 및 도구 사용
- 테스팅 패턴에 맞는 테스트 작성
- 적절하게 포맷된 pull request 생성

Custom rules가 없으면 Open SWE는 프로젝트와 맞지 않을 수 있는 일반적인 모범 사례를 사용합니다.

## 지원되는 섹션

### `<general_rules>`

프로젝트별 코딩 표준 및 규칙:

- 패키지 매니저 선호도 (예: "항상 Yarn 사용")
- 코드 스타일 요구사항
- Import/export 패턴
- 아키텍처 가이드라인
- 일반적인 스크립트 및 실행 방법

예시: "테스트를 실행하려면 `yarn test`를 실행하세요", 또는 "프로젝트를 빌드하려면 `yarn build`를 실행하세요"

### `<repository_structure>`

코드베이스 구조에 대한 설명:

- Monorepo vs 단일 패키지 구조
- 주요 디렉토리 및 그 목적
- 패키지 관계 및 의존성
- 빌드 시스템 구성

이 섹션에는 새 파일, 앱, 패키지 등을 어디에/어떻게 생성할지에 대한 컨텍스트를 포함하세요.

예시: "새 패키지를 생성할 때는 `packages` 디렉토리 안에 배치하세요."

### `<dependencies_and_installation>`

패키지 관리 및 설정 지침:

- 패키지 매니저 명령어
- 설치 절차
- 주요 의존성 및 그 목적
- Workspace 구성 세부사항

이 섹션에는 프로젝트에서 사용되는 패키지 매니저와 의존성을 설치하는 방법/위치에 대한 정보를 포함해야 합니다.

예시: "resolution을 추가하는 경우가 아니라면 루트 `package.json`이 아닌 사용되는 특정 앱/패키지 내부에 항상 의존성을 설치하세요."

### `<testing_instructions>`

테스팅 프레임워크 및 관행:

- Test runner 구성 (Jest, Vitest 등)
- 테스트 파일 명명 규칙
- 다양한 테스트 유형 실행 방법
- 프로젝트의 테스팅 모범 사례 (예: 작성할 테스트 유형)

예시: "변경 후 항상 `yarn test`를 실행하세요."

### `<pull_request_formatting>`

PR 생성 및 포맷 가이드라인:

- 제목 및 설명 템플릿
- 필수 섹션 또는 체크리스트
- 리뷰 프로세스 요구사항
- 링크 규칙

예시: "Pull request 본문에는 변경 사항을 테스트하는 방법을 설명하는 글머리 기호가 포함된 `Testing Steps` 섹션이 포함되어야 합니다."

## 파일 이름

Open SWE는 다음 파일에서 custom rules를 읽습니다 (우선순위 순서):

1. `AGENTS.md` (권장)
2. `AGENT.md`
3. `CLAUDE.md`
4. `CURSOR.md`

<Note>
  프로젝트 간 일관성을 위해 `AGENTS.md`를 표준 파일명으로 사용하세요.
</Note>

## 누락된 섹션

Custom rules 파일에 XML 섹션 태그가 포함되어 있지 않으면 전체 파일 내용이 custom rules 섹션 내부의 프롬프트로 전달됩니다.

## 포맷 예시

Custom rules 파일에 적절한 XML 섹션 태그가 포함되어 있으면 알려진 태그 내부의 내용만 시스템 프롬프트로 전달됩니다.

알려진 태그:
- `<general_rules>`
- `<repository_structure>`
- `<dependencies_and_installation>`
- `<testing_instructions>`
- `<pull_request_formatting>`

각 태그는 적절한 닫는 태그를 _반드시_ 포함해야 합니다. 태그에 닫는 태그가 없으면 전체 파일 내용이 custom rules 섹션 내부의 프롬프트로 전달됩니다.

다음은 `AGENTS.md` 파일이 어떻게 보여야 하는지를 보여주는 예시입니다:

```markdown
<general_rules>

- Always use Yarn as the package manager
- Follow strict TypeScript practices
- Use ESLint and Prettier for code quality
  </general_rules>

<repository_structure>
This is a Yarn workspace monorepo with three main packages:

- apps/web: Next.js frontend
- apps/api: Express backend
- packages/shared: Common utilities
  </repository_structure>

<dependencies_and_installation>
Run `yarn install` from the repository root to install all dependencies.
Key dependencies include React 18, TypeScript, and Jest for testing.
</dependencies_and_installation>

<testing_instructions>

- Run `yarn test` for unit tests
- Run `yarn test:e2e` for end-to-end tests
- Test files use `.test.ts` extension
- Use Jest with React Testing Library
  </testing_instructions>

<pull_request_formatting>
PR titles should follow: "feat: description" or "fix: description"
Include a brief description and link any related issues.
</pull_request_formatting>
```
